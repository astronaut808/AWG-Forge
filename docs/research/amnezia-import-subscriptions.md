# AmneziaVPN Import And Subscription Research

Status: research note, not an implementation plan.

Date: 2026-05-22.

## Scope

This document answers one product question:

Can awg-forge provide a subscription or auto-updating AmneziaVPN import flow so users receive changed endpoints/configs without manually downloading a fresh `.conf` file?

The short answer is: not safely as a production feature yet.

awg-forge should keep `.conf` download/import as the supported path. A future `vpn://` or native Amnezia import flow can be explored behind an experimental flag, but only after real-device tests confirm behavior across current AmneziaVPN desktop, iOS, and Android builds.

## Current awg-forge Baseline

awg-forge currently supports:

- rendering AmneziaWG Legacy / 1.0, 1.5-oriented, and 2.0 `.conf` files;
- client config download as the primary import path;
- stale config detection when tunnel settings change;
- server-host override per tunnel;
- encrypted backup/restore and restore dry-run;
- support bundles without secrets.

QR import is intentionally not exposed because plain `.conf` import proved more reliable across current AmneziaVPN clients.

## What AmneziaVPN Imports

Official Amnezia docs list `.vpn`, `.ovpn`, `.conf`, and selected `.json` formats, and describe `.conf`/`.json` file import as a normal connection setup path. The Amnezia client import controller also accepts several shapes:

- native Amnezia JSON objects;
- `vpn://` strings containing base64-url encoded, optionally compressed JSON;
- plain WireGuard/AmneziaWG configs with `[Interface]` and `[Peer]`;
- several non-AWG URI schemes such as VLESS, VMess, Trojan, and Shadowsocks.

For awg-forge, the important part is that plain `.conf` is a first-class import path. The client source detects a WireGuard/AWG config by `[Interface]` and `[Peer]` sections, then maps the parsed keys into an internal Amnezia JSON server object.

The same source also detects AWG versions from fields:

- Legacy/AWG detection requires the core junk and header fields.
- AWG 1.5 is detected when special junk fields `I1-I5` are present without the 2.0-only `S3/S4` pair.
- AWG 2.0 is detected when the config has both `S3` and `S4`.

That means awg-forge's `.conf` rendering remains aligned with the AmneziaVPN import path.

## What `vpn://` Means

In Amnezia client self-hosted export code, `vpn://` is generated by:

1. converting an internal Amnezia server JSON object to JSON bytes;
2. compressing the bytes with Qt compression;
3. base64-url encoding the result without trailing equals;
4. prefixing it with `vpn://`.

In import code, Amnezia reverses the same process: remove `vpn://`, base64-url decode, try to decompress, then parse JSON and classify it as an Amnezia config.

This is not a generic WireGuard subscription format. It is an Amnezia application payload format.

## What Subscriptions Mean In AmneziaVPN

The subscription code path in AmneziaVPN appears tied to Amnezia API/Gateway services. It is not just “download a config from any URL and refresh it”.

The client subscription controller:

- generates client-side protocol data, including an AWG/WireGuard key pair for the current client;
- sends API payloads to gateway endpoints such as `v1/config`, `v1/subscriptions`, `v1/native_config`, and revoke endpoints;
- stores `api_config`, `auth_data`, service type, service protocol, country metadata, and a stored VPN key;
- can update API-backed configs through `updateServiceFromGateway`;
- can export native configs through the gateway path.

For awg-forge, this matters because there is no evidence in the public client code that a random self-hosted URL can be registered as an auto-refresh subscription with the same behavior as the Amnezia Gateway service.

## Difference Between `.conf`, `vpn://`, And A Subscription

| Path | What it is | Production status for awg-forge |
| --- | --- | --- |
| `.conf` download | Native WireGuard/AmneziaWG INI config imported by AmneziaVPN | Supported and recommended |
| Plain QR with `.conf` text | QR-encoded config text | Not exposed; was unreliable on iOS in testing |
| `vpn://...` | Amnezia-native compressed JSON import payload | Research-only; schema and platform behavior need validation |
| Subscription/update | API-backed refresh/reissue flow in AmneziaVPN code | Not currently suitable for awg-forge without implementing a compatible API contract |

## Product Analysis

The desired user experience is valid:

- admin changes a tunnel endpoint or parameters;
- users do not need to manually receive a new file;
- clients update safely.

But the risk is high if we approximate this with an undocumented payload:

- native Amnezia JSON schema is internal app behavior, not a documented stable public API;
- update behavior appears coupled to Amnezia Gateway fields and API endpoints;
- awg-forge would need to expose private client configs through a network endpoint;
- revocation, expiration, audit, rate limiting, and token rotation become security-critical;
- different AmneziaVPN platforms may parse or refresh differently;
- routers and non-Amnezia clients still need `.conf`.

## Security Requirements If Implemented Later

Any future subscription/import endpoint must be designed as secret distribution, not as a convenience URL.

Minimum requirements:

- per-client random token with at least 128 bits of entropy;
- token stored hashed in state, not plaintext;
- optional token expiration and manual rotation;
- immediate token revocation when a client is deleted;
- no shared tunnel-wide subscription URL;
- TLS strongly recommended for public access;
- no secrets in access logs, errors, support bundles, or doctor output;
- rate limiting for config delivery endpoints;
- no directory listing or predictable URL paths;
- response headers disabling caching by shared proxies;
- clear UI warning that the URL grants access to the client private key;
- explicit compatibility label: AmneziaVPN only, not routers, not generic WireGuard clients.

## Architecture If We Explore It

A safe experimental design would be:

1. Keep `.conf` as canonical rendered artifact.
2. Add optional per-client `import_token_hash`, `import_token_created_at`, `import_token_last_used_at`, and `import_token_revoked_at`.
3. Add a public read-only endpoint like `/s/<token>` that returns either:
   - plain `.conf` with `text/plain`, or
   - an Amnezia `vpn://` payload only when the feature flag is enabled.
4. Add UI actions:
   - “Generate import link”;
   - “Rotate import link”;
   - “Revoke import link”;
   - “Download .conf”.
5. Log only token hash prefix and client ID, never the config.
6. Add integration tests that decode generated `vpn://` payloads and compare them with the expected native Amnezia JSON shape.
7. Add real-device acceptance tests for desktop, iOS, and Android AmneziaVPN.

## Open Questions

These must be answered before implementation:

- Does current AmneziaVPN support adding a self-hosted non-Gateway subscription URL?
- Can a `vpn://` payload update an existing imported profile, or does it always add a new profile?
- Does AmneziaVPN ever poll/update a self-hosted `vpn://` payload without user action?
- What exact native JSON schema is required for AWG 2.0 on all platforms?
- Does iOS handle `vpn://` links from Safari/Mail/Notes differently than QR or file import?
- Can endpoint-only changes be applied without rotating keys?
- How does AmneziaVPN resolve duplicate configs: by CRC, host, public key, stored VPN key, or another field?

## Recommendation

Do not implement subscription or `vpn://` import as a production feature in v0.6.x/v0.7.x.

Recommended roadmap:

1. Keep `.conf` download/import as the only stable production flow.
2. Improve stale config UX: make it very clear which clients need a new `.conf`.
3. Add a safe “reissue config” flow that downloads fresh `.conf` after endpoint/protocol changes.
4. Later, create a separate experimental branch for Amnezia native import research.
5. Only promote native import/subscription after source-level compatibility is proven and real devices pass.

## Decision For awg-forge

Current decision:

- `.conf` remains canonical.
- `vpn://` and subscription support are not part of the stable product.
- awg-forge may expose an authenticated experimental `Import key` action for AmneziaVPN / DefaultVPN testing. This key is not a subscription endpoint and should not be treated as router or production-client support.
- Any future Amnezia-native import should be explicitly experimental and documented as AmneziaVPN-specific.

This keeps awg-forge predictable for routers, desktop clients, iOS, Android, and manual production operations.
